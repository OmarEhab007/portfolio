# Building an Air-Gapped RAG Agent for BMC Remedy with LangChain4j and pgvector

*March 10, 2026*

---

## Introduction

After years of administering BMC Remedy, I watched the same pattern repeat across every support team I worked with: a new incident comes in, the engineer searches for keywords, scrolls through hundreds of tickets, and eventually either finds a vaguely relevant resolution or starts from scratch. All that institutional knowledge — thousands of resolved incidents — was sitting in the database, practically inaccessible.

I built the BMC Remedy RAG Agent to change that. It's an enterprise AI assistant that turns years of IT support history into instant, context-aware answers. Ask it "how do we fix VPN timeout errors on Windows 11?" and it pulls the most semantically relevant past incidents, feeds them to a local LLM, and gives you a synthesized answer with source citations pointing to specific incident numbers. The entire system runs on-premise with zero cloud dependencies — a hard requirement when you're handling sensitive IT operations data.

This post is the technical deep-dive: how I designed the RAG pipeline, why I chose LangChain4j with Spring Boot, how pgvector handles hybrid search, and what it takes to make agentic AI operations work in an air-gapped enterprise environment.

## RAG Architecture Overview

The system is a multi-module Maven project with six modules, each with a clear responsibility:

```
User → Web Chat (React 19) → API Gateway (8080) → RAG Service → Vector Store → Ollama LLM
                                    │                    │              │
                                    │                    │              ├─ pgvector (HNSW)
                                    │                    │              └─ Full-text search
                                    │                    │
                                    │                    ├─ LangChain4j orchestration
                                    │                    ├─ ReBAC filtering
                                    │                    └─ Agentic tools
                                    │
                                    └─ OAuth2/JWT security, WebSocket streaming
```

- **api-gateway** — REST controllers, WebSocket streaming for token-by-token responses, OAuth2/JWT security
- **rag-service** — the brain: LangChain4j orchestration, chat memory, ReBAC-aware retrieval, agentic tool definitions
- **vectorization-engine** — ONNX-based local embeddings using all-MiniLM-L6-v2 (384 dimensions), semantic chunking with Apache Tika for document extraction
- **vector-store** — PostgreSQL 16 + pgvector with HNSW indexing, hybrid search combining vector similarity and full-text ranking
- **remedy-connector** — native BMC AR Java API integration with a ThreadLocal connection pool
- **frontend/web-chat** — React 19 + TypeScript + Tailwind CSS, bilingual (English/Arabic with RTL)

The key design principle: every piece of intelligence runs locally. Embeddings are generated by an ONNX model loaded in-process. The LLM runs through Ollama on the same network. No API keys, no cloud endpoints, no data leaving the perimeter.

## LangChain4j + Spring Boot: The Orchestration Layer

I chose LangChain4j 0.35 over rolling my own RAG pipeline. The framework has matured significantly — its Spring Boot integration gives you declarative configuration for the entire AI stack.

Here's the core RAG service configuration:

```java
@Configuration
public class RagConfiguration {

    @Bean
    public ChatLanguageModel chatModel(OllamaProperties props) {
        return OllamaChatModel.builder()
                .baseUrl(props.getBaseUrl())
                .modelName(props.getModel())
                .temperature(0.2)
                .timeout(Duration.ofMinutes(3))
                .build();
    }

    @Bean
    public EmbeddingModel embeddingModel() {
        return new OnnxEmbeddingModel(
                "models/all-minilm-l6-v2.onnx",
                "models/tokenizer.json"
        );
    }

    @Bean
    public ContentRetriever contentRetriever(
            EmbeddingModel embeddingModel,
            EmbeddingStore<TextSegment> embeddingStore) {
        return EmbeddingStoreContentRetriever.builder()
                .embeddingModel(embeddingModel)
                .embeddingStore(embeddingStore)
                .maxResults(8)
                .minScore(0.65)
                .build();
    }

    @Bean
    public RetrievalAugmentor retrievalAugmentor(
            ContentRetriever contentRetriever,
            ReBACFilter reBACFilter) {
        return DefaultRetrievalAugmentor.builder()
                .contentRetriever(contentRetriever)
                .queryTransformer(new CompressingQueryTransformer())
                .contentInjector(new DefaultContentInjector())
                .build();
    }
}
```

The `CompressingQueryTransformer` is worth calling out. In a multi-turn conversation, the user might say "what about for Linux?" after asking about VPN issues. The transformer takes the chat history and rewrites the query into a self-contained search: "how do we fix VPN timeout errors on Linux?" This dramatically improves retrieval quality in conversational contexts.

Chat memory is managed per-session with a sliding window. LangChain4j's `MessageWindowChatMemory` keeps the last N messages, which prevents context windows from blowing up on long conversations while keeping enough history for coherent multi-turn dialogue.

## Local Embeddings with ONNX

Running embeddings locally was non-negotiable for air-gapped deployment. I use `all-MiniLM-L6-v2` exported to ONNX format — it produces 384-dimensional vectors and runs inference in single-digit milliseconds per chunk on commodity hardware.

The vectorization engine handles document ingestion in three stages:

1. **Extraction** — Apache Tika pulls text from incident records, attachments (PDFs, Word docs), and work log entries
2. **Chunking** — semantic chunking splits text on paragraph boundaries with overlap, keeping each chunk under 512 tokens to stay within the embedding model's sweet spot
3. **Embedding** — the ONNX model generates vectors in-process, no network calls

The chunking strategy matters more than people think. I experimented with fixed-size chunking (every 256 tokens) and it was noticeably worse for retrieval. Incident tickets have structure — summary, description, resolution notes, work log entries — and splitting mid-sentence across those boundaries loses context. Semantic chunking respects those boundaries.

## pgvector + Hybrid Search: Getting Retrieval Right

Pure vector similarity search has a well-known weakness: it can miss exact matches. If someone searches for "INC00045821," a semantic search might return incidents about similar topics but miss the exact ticket. That's unacceptable in an IT support context.

I implemented hybrid search combining pgvector's vector similarity with PostgreSQL's built-in full-text search, fused using Reciprocal Rank Fusion (RRF):

```sql
-- Hybrid search: vector similarity + full-text, merged via RRF
WITH vector_results AS (
    SELECT id, content, incident_number,
           1 - (embedding <=> $1::vector) AS vector_score,
           ROW_NUMBER() OVER (ORDER BY embedding <=> $1::vector) AS vector_rank
    FROM incident_chunks
    WHERE support_group_id = ANY($3)  -- ReBAC filter
    ORDER BY embedding <=> $1::vector
    LIMIT 20
),
text_results AS (
    SELECT id, content, incident_number,
           ts_rank_cd(search_vector, plainto_tsquery('english', $2)) AS text_score,
           ROW_NUMBER() OVER (
               ORDER BY ts_rank_cd(search_vector, plainto_tsquery('english', $2)) DESC
           ) AS text_rank
    FROM incident_chunks
    WHERE search_vector @@ plainto_tsquery('english', $2)
      AND support_group_id = ANY($3)  -- ReBAC filter
    LIMIT 20
),
fused AS (
    SELECT COALESCE(v.id, t.id) AS id,
           COALESCE(v.content, t.content) AS content,
           COALESCE(v.incident_number, t.incident_number) AS incident_number,
           COALESCE(1.0 / (60 + v.vector_rank), 0) +
           COALESCE(1.0 / (60 + t.text_rank), 0) AS rrf_score
    FROM vector_results v
    FULL OUTER JOIN text_results t ON v.id = t.id
)
SELECT id, content, incident_number, rrf_score
FROM fused
ORDER BY rrf_score DESC
LIMIT 8;
```

The `60` constant in the RRF formula is standard — it prevents high-ranked results from dominating the fusion. The HNSW index on the embedding column makes the vector search portion fast even at scale:

```sql
CREATE INDEX idx_incident_chunks_embedding
    ON incident_chunks
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 200);
```

I tuned `m = 16` and `ef_construction = 200` after benchmarking with production-scale data. Higher values improve recall at the cost of index build time and memory. For our dataset size (hundreds of thousands of chunks), these values hit the sweet spot.

Flyway manages all schema migrations, including the pgvector extension setup and index creation. This keeps the database schema version-controlled and reproducible across environments.

## ReBAC Security: Users Only See What They Should

In an enterprise Remedy environment, not everyone should see every incident. Support groups have boundaries. A network operations engineer shouldn't be searching through HR system incidents.

I implemented Relationship-Based Access Control (ReBAC) at the retrieval layer. Every incident chunk is tagged with its source support group ID during ingestion. At query time, the user's authorized groups are resolved from their JWT token and injected as a filter directly into the SQL query — the `WHERE support_group_id = ANY($3)` clause you see above.

This is not post-retrieval filtering. The filter is pushed down into both the vector and full-text search branches, which means unauthorized chunks never leave the database. This matters for performance (we're not retrieving and discarding results) and for security (there's no window where unauthorized data exists in application memory).

```java
@Service
public class ReBACFilter {

    public List<String> resolveAuthorizedGroups(Authentication auth) {
        var jwt = (JwtAuthenticationToken) auth;
        var groups = jwt.getToken().getClaimAsStringList("support_groups");
        return groups != null ? groups : List.of();
    }
}
```

## Agentic Operations: AI That Can Act

The RAG agent doesn't just answer questions — it can create and update incidents through a confirmation workflow. This is where LangChain4j's tool-calling support shines.

I defined tools that the LLM can invoke:

```java
@Tool("Create a new incident in BMC Remedy")
public ToolResult createIncident(
        @P("Short description of the incident") String summary,
        @P("Detailed description") String description,
        @P("Priority: Critical, High, Medium, Low") String priority,
        @P("Customer name or ID") String customer) {

    // Duplicate detection before creation
    var duplicates = searchService.findSimilar(summary, 0.85);
    if (!duplicates.isEmpty()) {
        return ToolResult.of(
            "Found %d potential duplicate(s): %s. Proceed anyway?"
                .formatted(duplicates.size(),
                    duplicates.stream()
                        .map(d -> d.getIncidentNumber())
                        .collect(joining(", "))));
    }

    var draft = IncidentDraft.builder()
            .summary(summary)
            .description(description)
            .priority(Priority.fromString(priority))
            .customer(customer)
            .build();

    // Return draft for user confirmation — never auto-submit
    return ToolResult.of(confirmationService.stage(draft));
}
```

Two things are critical here. First, **duplicate detection**: before creating an incident, the agent runs a similarity search against existing incidents. If something similar exists above a 0.85 threshold, it surfaces the potential duplicates and asks the user. This prevents the single most common IT support waste — duplicate tickets.

Second, **confirmation workflow**: the agent never creates an incident autonomously. It stages a draft and asks the user to confirm. This is a deliberate design choice. In an enterprise environment, an AI that silently creates or modifies production data would be a non-starter.

Rate limiting adds another safety layer: 100 chat requests per minute and 10 action operations per hour per user. These limits were calibrated from observed usage patterns.

## Air-Gapped Deployment

The entire stack runs without internet access. Here's what that means in practice:

- **LLM** — Ollama runs locally with models pre-downloaded and baked into the deployment artifact
- **Embeddings** — ONNX model files ship with the application JAR
- **Database** — PostgreSQL with pgvector, no managed cloud database
- **Frontend** — all assets bundled, no CDN dependencies
- **Authentication** — on-premise OAuth2 provider, no Auth0 or Okta cloud calls

The Docker Compose and Kubernetes manifests define the complete stack. The only external dependency at runtime is the on-premise Remedy AR Server that supplies the incident data.

Incremental sync keeps the knowledge base current without full re-ingestion. The remedy-connector uses CDC-based polling — it queries Remedy for incidents modified since the last sync timestamp, vectorizes the changes, and upserts the chunks. This runs on a configurable schedule and typically processes a day's worth of changes in seconds.

## Lessons Learned

**Hybrid search is not optional for enterprise RAG.** Pure vector search failed on exact ticket number lookups, specific error codes, and server hostnames. Adding full-text search with RRF fusion solved all of these without degrading semantic search quality.

**Chunking strategy is your biggest retrieval lever.** I spent more time tuning chunking than any other component. The final approach — semantic chunking that respects incident field boundaries with 50-token overlap — outperformed every fixed-size strategy I tested.

**ReBAC must be at the database layer.** Early prototypes did post-retrieval filtering in Java. This was both slower (fetching results only to discard them) and riskier (a bug could leak unauthorized data into the LLM context). Pushing filters into the SQL query is the right answer.

**Confirmation workflows are essential for agentic AI.** The first version of the incident creation tool auto-submitted. It took exactly one duplicate ticket created in a demo to convince me that staging + confirmation is the only acceptable pattern for production write operations.

**Air-gapped deployment forces better architecture.** Not being able to call a cloud API for embeddings or inference means you own every piece of the stack. That's more work upfront, but the result is a system with no external failure modes, predictable latency, and zero ongoing cloud costs.

*Build intelligence into your infrastructure, not around it.* ☾
